## Resource 1: Ultraplan — Plan Complex Tasks in the Cloud While Your Terminal Stays Free

**Source:** [Plan in the cloud with Ultraplan — Claude Code Docs](https://code.claude.com/docs/en/ultraplan)
**Also:** [Claude Code's Ultraplan Bridges the Gap Between Planning and Execution — DevOps.com](https://devops.com/claude-codes-ultraplan-bridges-the-gap-between-planning-and-execution/)
**Also:** [Claude Code Ultraplan: planning in the cloud, executing wherever you want — Medium](https://medium.com/arckit/claude-code-ultraplan-planning-in-the-cloud-executing-wherever-you-want-f4112ccd5e3c)
**Time:** ~5 min (skim the official docs + one overview article)
**Category:** New Feature — Cloud Planning & Collaboration

### Why This Is Relevant

Ultraplan entered early preview during Week 15 (April 6-10) and is the most significant new feature you haven't explored yet. Here's what it does: you start a planning job from your terminal, Claude drafts the plan on Anthropic's Cloud Container Runtime using Opus 4.6 with up to 30 minutes of reserved compute, and you review/comment on it via the Claude Code web interface — all while your terminal stays free for other work.

**Why this matters for your workflow:**

Your reverse engineering pipeline currently runs sequentially in your terminal. When you're planning a large analysis — "map all 47 microservices, trace the 12 critical API flows, identify the 3 highest-risk coupling points" — that planning phase blocks everything else. Ultraplan moves that entire planning phase to the cloud.

**The collaboration angle is key for ISP:**

Unlike local Plan Mode, Ultraplan's web editor supports inline comments on specific plan sections, emoji reactions to flag approval or revision, and a structured outline sidebar. For a consulting team, this means the tech lead can draft the reverse engineering plan, the architect can comment on the service decomposition approach, and the project manager can flag timeline risks — all before a single line of analysis runs. This is the "collaborative AI planning" slide your ISP presentation is missing.

**How it works:**

1. In your terminal: `/ultraplan Analyze the ISP codebase: map all microservices, trace critical API flows, identify coupling hotspots`
2. Claude drafts the plan in the cloud (your terminal is free)
3. Open the web editor link, review sections, leave inline comments
4. Execute the approved plan in the browser or pull it back to your terminal

**Requirements:** Claude Code web account, GitHub repo, v2.1.91+. Does not work with Bedrock, Vertex AI, or Foundry.

### Key Takeaway

**Ultraplan turns your local reverse engineering workflow into a collaborative cloud-based planning process.** For ISP, this is how you scale from "one consultant runs the analysis" to "the entire team reviews and approves the analysis plan before execution." Try it today on a subset of the ISP codebase — plan the analysis of one service cluster in the cloud while you continue other work in your terminal.

## Resource 3: Managed Agents Production Cookbook — From Prototype to Deployed Service

**Source:** [Managed Agents tutorial: production setup — Claude Cookbook](https://platform.claude.com/cookbook/managed-agents-cma-operate-in-production)
**Also:** [Authenticate with vaults — Claude API Docs](https://platform.claude.com/docs/en/managed-agents/vaults)
**Also:** [Managed Agents tutorial: iterate on a failing test suite — Claude Cookbook](https://platform.claude.com/cookbook/managed-agents-cma-iterate-fix-failing-tests)
**Time:** ~5 min (skim the production setup cookbook + vault docs)
**Category:** Production Deployment — Hands-On Tutorial

### Why This Is Relevant

On April 14 (digest #24), you learned *what* Managed Agents is and *why* it matters for productizing your reverse engineering workflow. Today's resource is the *how*: the official Claude Cookbook's end-to-end production setup tutorial that walks you through actually deploying an agent.

**What the cookbook covers:**

1. **Vault-backed credentials**: Register your GitHub/Jira/Slack tokens once in a vault, reference them by ID at session creation. No secrets in code, no token management headaches. Per-end-user credential isolation means each ISP developer's access stays separate.

2. **Session webhooks for human-in-the-loop**: The `session.status_idled` webhook pattern lets you build approval workflows without holding a long-lived HTTP connection. The agent works, idles when it needs human input, fires a webhook, and resumes when approved. This is how you build "AI proposes the architecture analysis plan, human approves, AI executes" — exactly your Ultraplan workflow but as a production service.

3. **Resource lifecycle CRUD**: Create, read, update, and archive agent sessions with proper state management. This is how you build a dashboard where ISP developers can see all running architecture analyses, their status, and their outputs.

4. **MCP toolsets**: Instead of building custom tools, connect your agent to SaaS APIs (GitHub, Jira, Confluence) via MCP servers. The agent talks to these services directly without round-tripping through your application.

**The path from your current workflow to a deployed service:**

```
Current: /reverse-engineer (local Claude Code skill)
    ↓
Step 1: Wrap it as a Managed Agent session with vault credentials
    ↓
Step 2: Add session.status_idled webhook for plan approval
    ↓
Step 3: Connect GitHub + Confluence MCP servers for direct repo access and doc publishing
    ↓
Step 4: Trigger via Routine (scheduled) or API (on-demand from ISP's internal tools)
```

**Cost estimate for ISP:** A weekly architecture analysis that runs for 2 hours = ~$0.16/week in runtime + ~$5-10 in tokens ≈ $45/month. That's negligible compared to the consulting hours it replaces.

### Key Takeaway

**The Cookbook's production tutorial is the bridge between your local prototype and a deployed ISP service.** Block 30 minutes this week to walk through it. Focus on the vault setup (you'll need GitHub credentials for repo access) and the webhook pattern (this is how you add human approval to the automated pipeline). By the end, you'll have a concrete architecture for the "productization" slide in your ISP presentation — not a concept, but a working deployment plan.

## Quick Win: Feed Your Reverse Engineering Output Back into CLAUDE.md (5 min)

Nick Tune's insight is immediately actionable. After your next `/reverse-engineer` run on any ISP service, take the top-level findings and add them to the repo's CLAUDE.md:

```bash
# In your ISP project directory
cat >> CLAUDE.md << 'EOF'

## Architecture Overview (auto-generated by /reverse-engineer)

### Service Boundaries
- [List the key services and their responsibilities]

### Critical API Flows
- [List the 3-5 most important end-to-end flows]

### Known Coupling Hotspots
- [List areas where services are tightly coupled]

### Data Flow Summary
- [Key data pipelines and their direction]

EOF
```

Then edit in the actual findings from your last analysis run. Every future Claude Code session in this repo will automatically have this architectural context, making every task — bug fixes, feature additions, code reviews — faster and more accurate.

**Why this is today's Quick Win:** It takes 5 minutes, it compounds on every future session, and it demonstrates the "living documentation" concept from Nick Tune's article — which is exactly the continuous value story you need for the ISP presentation. The architecture doc isn't a one-time deliverable; it's a self-improving knowledge base that makes the AI more effective over time.
