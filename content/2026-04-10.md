## Resource 1: /batch — Parallel Codebase-Wide Transformations

**Source:** [Claude Code /simplify and /batch Commands Guide — claudefa.st](https://claudefa.st/blog/guide/mechanics/simplify-batch-commands)
**Also:** [Claude Code Batch Processing Complete Guide — SmartScope](https://smartscope.blog/en/generative-ai/claude/claude-code-batch-processing/)
**Time:** ~5 min read
**Category:** Large Codebase Operations & Orchestration

### Why This Is Relevant
You've been using subagents (April 9) and Plan Mode (April 8) to approach the 100K-line Java/Python reverse engineering project. `/batch` is the next level — it orchestrates codebase-wide transformations by decomposing your instruction into 5-30 independent worker agents, each running in its own git worktree, each producing its own PR.

**How `/batch` works — three phases:**

1. **Research & Planning**: The orchestrator enters Plan Mode automatically, launches Explore agents to investigate the codebase, identifies target files and patterns, then decomposes the work into independent units. Each unit must be independently mergeable with no cross-dependencies.

2. **Parallel Execution**: Each worker agent gets a fully self-contained prompt with the overall goal, its specific task, and codebase conventions discovered during research. After implementing, each worker runs `/simplify` on its own changes, executes the test suite, commits, pushes, and opens a PR. Worktree isolation means agents never step on each other's changes.

3. **Progress Tracking**: The orchestrator renders a live status table, updating as agents complete, pulling PR URLs from each agent's output. Final summary: "22/24 units landed as PRs."

**For the 100K-line Java/Python project, try these:**

```
/batch document all REST API endpoints in src/main/java with OpenAPI-style annotations

/batch add type hints to all Python functions in src/python that currently have none

/batch identify and flag all database queries that don't use parameterized inputs
```

Each of these would be days of manual work. `/batch` decomposes them into parallel units and produces individually reviewable PRs.

**When NOT to use `/batch`:** If changes are interdependent (e.g., a refactor where module A's new interface changes module B's imports), use a standard Claude Code session or Agent Teams (Resource 2) instead. `/batch` requires each unit to be independently mergeable.

### Key Takeaway
**`/batch` is the production-grade version of the subagent pattern from April 9.** Instead of manually defining subagents for each part of the codebase, `/batch` automatically decomposes the work, assigns workers, and produces PRs. For the reverse engineering project, combine the `codebase-archaeologist` subagent from yesterday's Quick Win with `/batch` to systematically document the entire codebase in parallel — what would take a team days happens in hours.

## Resource 2: Agent Teams — Multi-Session Orchestration

**Source:** [Orchestrate Teams of Claude Code Sessions — Claude Code Docs](https://code.claude.com/docs/en/agent-teams)
**Also:** [Claude Code Agent Teams: Setup & Usage Guide 2026 — claudefa.st](https://claudefa.st/blog/guide/agents/agent-teams)
**Time:** ~5 min read
**Category:** Team Collaboration & Parallel Development

### Why This Is Relevant
Subagents (April 9) work inside a single session. `/batch` (Resource 1) orchestrates independent parallel units. Agent Teams fill the gap between them — multiple Claude Code instances that can **communicate with each other**, share a task list, and coordinate on work that requires collaboration. This is the tool for the ISP Digital Twin project where frontend, backend, and testing need to stay in sync.

**Key differences from subagents:**

| Feature | Subagents | Agent Teams |
|---|---|---|
| Context window | Shared (within parent session) | Independent (each teammate has its own) |
| Communication | Report back to parent only | Message each other directly |
| User interaction | Only through parent | You can talk to any teammate |
| Coordination | Parent orchestrates | Shared task list + peer messaging |
| Cost | 1x (same session) | 3-7x tokens (multiple sessions) |

**The strongest use cases for your work:**

1. **Research with competing hypotheses**: For the reverse engineering project, spin up 3 teammates — one analyzing the Java layer, one on Python, one on the integration points. They share findings and challenge each other's understanding.

2. **Cross-layer development**: For ISP Digital Twin, have one teammate on the frontend, one on the backend API, one writing tests. They coordinate through the shared task list so interfaces stay consistent.

3. **Parallel debugging**: When UAT surfaces a bug, have one teammate reproduce it, another trace the root cause in code, and a third check related systems for similar issues.

**Setup (1 minute):**
Add to your `.claude/settings.json`:
```json
{
  "experimental": {
    "agentTeams": true
  }
}
```

Then start with: *"Create a team of 3 to analyze the Java API layer, the Python data processing layer, and the integration points between them. Share findings as you go."*

**Start small:** 3-5 teammates is the sweet spot. More adds coordination overhead without proportional benefit.

### Key Takeaway
**Agent Teams are for work that requires collaboration, not just parallelism.** Use `/batch` when each unit is independent (e.g., "add docs to every endpoint"). Use Agent Teams when teammates need to share context and challenge each other (e.g., "analyze how the Java and Python layers interact"). For the ISP reverse engineering project, Agent Teams let you run a multi-perspective architecture analysis where each agent's findings inform the others — closer to how a real consulting team would staff a codebase review.
