## Resource 1: Reverse Engineering Software Architecture with Claude Code

**Source:** [Reverse Engineering Your Software Architecture with Claude Code to Help Claude Code — Nick Tune (O'Reilly Radar)](https://www.oreilly.com/radar/reverse-engineering-your-software-architecture-with-claude-code-to-help-claude-code/)
**Also on:** [Medium](https://medium.com/nick-tune-tech-strategy-blog/reverse-engineering-your-software-architecture-with-claude-code-to-help-claude-code-1746a7b941bc)
**Time:** ~6 min read
**Category:** Codebase Analysis & Architecture Documentation

### Why This Is Relevant
This is directly applicable to the reverse engineering project you're advising on — the ~100K lines of Java/Python where the team needs architecture docs, API flows, and data structures. Nick Tune's approach uses Claude Code to map end-to-end flows across multiple repositories, which is exactly the task at hand.

**The workflow:**

1. **Give Claude access to all repositories**: In `.claude/settings.local.json`, grant permissions to read all local repos and use the GitHub CLI for any repos not available locally. This lets Claude traverse the entire system.

2. **Map end-to-end flows first**: Rather than documenting bottom-up (file by file), ask Claude to trace end-to-end business flows across repositories. This mirrors how stakeholders think about systems and produces immediately useful documentation.

3. **Use horizontal flow diagrams, not sequence diagrams**: Nick found that sequence diagrams become unreadable for complex multi-repo flows. Instead, use `flowchart LR` (left-to-right) Mermaid diagrams with each repository as a horizontal swimlane. Claude generates these natively.

4. **Validate completeness**: Ask Claude to list all API endpoints and events produced/consumed by each application. Cross-reference against the mapped flows to identify anything not yet documented — this catches orphaned services and hidden dependencies.

5. **Build a system overview file**: Create a bullet list of repository names with 1-2 sentences describing their relationships. This becomes the persistent context (`CLAUDE.md`) that makes all subsequent analysis faster and more accurate.

**For the 100K-line Java/Python project specifically:** Start with Plan Mode (Shift+Tab), have Claude explore the codebase structure without changing anything, then ask it to identify the top 5-10 end-to-end flows. This produces a skeleton architecture document in under an hour that would take a human team days.

### Key Takeaway
**Don't reverse-engineer bottom-up (class by class) — reverse-engineer top-down (flow by flow).** Ask Claude to trace business processes across the codebase and generate horizontal flow diagrams. This produces documentation that stakeholders can actually use, and it's the approach you can recommend directly in the reverse engineering email thread.

## Resource 2: 100+ Specialized Subagents for Codebase Analysis

**Source:** [awesome-claude-code-subagents — VoltAgent (GitHub)](https://github.com/VoltAgent/awesome-claude-code-subagents)
**Time:** ~4 min read + setup
**Category:** Large Codebase Analysis & Documentation

### Why This Is Relevant
When analyzing a 100K-line codebase, context window management becomes critical. Subagents solve this by running in isolated context windows — each one reads, analyzes, and reports back a summary without polluting your main session. This repo provides 100+ pre-built subagent templates organized by use case, several of which are directly applicable to your current projects.

**The 6 most relevant subagents for your work:**

| Subagent | What It Does | Use Case |
|---|---|---|
| `architect-reviewer` | Evaluates system design, structure, and architectural patterns | Reverse engineering the Java/Python architecture |
| `documentation-engineer` | Creates comprehensive documentation from existing code | Generating the missing API/data structure docs |
| `legacy-modernizer` | Analyzes older codebases for improvement opportunities | Assessing the takeover codebase |
| `java-architect` | Java-specific deep framework knowledge | Java portion of the 100K-line codebase |
| `python-pro` | Python-specific analysis and best practices | Python portion of the codebase |
| `code-reviewer` | Reviews code for patterns, best practices, and issues | Hackathon code review tasks |

**How to set up (2 minutes):**
Install the plugin: `/plugin marketplace add https://github.com/VoltAgent/awesome-claude-code-subagents`, then install the relevant packs (`voltagent-qa-sec` for reviewers, `voltagent-dev-exp` for documentation/modernization, `voltagent-lang` for language specialists).

**The key pattern for large codebases:**
```
Use subagents to analyze the Java modules in /src/main/java.
Have the architect-reviewer map the dependency graph,
the documentation-engineer generate API docs for each service,
and the java-architect identify framework patterns and anti-patterns.
Report findings in a single summary.
```

Each subagent runs in its own context window, reads hundreds of files, and returns only a summary. Your main session stays clean and focused on synthesizing findings.

### Key Takeaway
**For the reverse engineering project, combine Nick Tune's top-down flow approach (Resource 1) with specialized subagents for deep analysis.** Use `architect-reviewer` to map the architecture, `documentation-engineer` to generate the missing docs, and language-specific agents (`java-architect`, `python-pro`) for framework-level analysis. This parallelized approach is how you handle 100K lines without hitting context limits.

## Resource 3: Claude SDLC Toolkit — Full Lifecycle Slash Commands

**Time:** ~4 min read + setup

It's both a reference for your ISP presentation and a tool you can use on your own projects.

**The 13 SDLC commands:**

| Phase | Command | What It Does |
|---|---|---|
| Analysis | `/sdlc:prime` | Analyzes the codebase and generates context |
| Specification | `/sdlc:feature` | Creates feature specifications |
| Specification | `/sdlc:bug` | Creates bug specifications |
| Specification | `/sdlc:chore` | Creates chore/maintenance specs |
| Implementation | `/sdlc:implement` | Executes work with tracking |
| Implementation | `/sdlc:tdd` | Test-driven development flow |

| Git | `/sdlc:pull_request` | Creates PRs from implementations |

| Utilities | `/sdlc:install` | Manages dependencies |

**Two execution modes:**
1. **Full Automation**: specification → branch → implementation → commit → pull request (for trusted workflows)

**Three specialized agents built in:**

- **Performance-Oracle**: Evaluates algorithmic efficiency and scalability
- **PR-Comment-Resolver**: Addresses reviewer feedback and updates code

**For the ISP presentation:** This toolkit demonstrates a concrete "GenAI-augmented SDLC" that you can reference. The Plan-Only mode maps directly to enterprise governance requirements.

## Quick Win: Draft a Reverse Engineering Subagent for the Java/Python Project (2 min)

Based on the reverse engineering email thread, here's a custom subagent you can create right now and share with the team evaluating the 100K-line codebase:

Create this file at `.claude/agents/codebase-archaeologist.md`:

```markdown
---
name: codebase-archaeologist
description: Reverse-engineers architecture, API flows, and data structures from legacy codebases
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior software architect specializing in legacy system analysis.
Your task is to reverse-engineer documentation from an existing codebase.

For each module or service you analyze:
1. Identify the primary responsibility and business domain
2. Map all API endpoints (REST, SOAP, message queues) with request/response schemas
3. Document data structures and their relationships (ERD-style)
4. Trace the top 5 business flows end-to-end using Mermaid flowchart LR diagrams
5. List all external dependencies and integration points
6. Flag architectural risks, anti-patterns, and technical debt

Output format: structured markdown with Mermaid diagrams.
Keep each repository as a horizontal swimlane in flow diagrams.
```

Then use it: *"Use the codebase-archaeologist subagent to analyze the /src directory and generate architecture documentation."*

This subagent runs in an isolated context window, so it can read hundreds of files without affecting your main session. Share the agent definition with the team — anyone with Claude Code can use it immediately on the target codebase.

**Why this compounds:** This subagent + Plan Mode (from April 8) + the flow-first approach from Resource 1 = a repeatable methodology for reverse engineering any codebase. This is the kind of reusable consulting asset that turns a one-off project into a capability.
