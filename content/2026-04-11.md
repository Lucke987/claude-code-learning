## Resource 1: Custom Skills — Turn Your Workflows Into Team-Shareable Slash Commands

**Source:** [Extend Claude with Skills — Claude Code Docs](https://code.claude.com/docs/en/skills)
**Also:** [How to Build Claude Code Skills That Actually Work — DEV Community](https://dev.to/whoffagents/how-to-build-claude-code-skills-custom-slash-commands-that-actually-work-1nje)
**Also:** [How to Build a Production-Ready Claude Code Skill — Towards Data Science](https://towardsdatascience.com/how-to-build-a-production-ready-claude-code-skill/)
**Time:** ~5 min read
**Category:** Workflow Packaging & Team Enablement

### Why This Is Relevant
Over the past 3 weeks, you've assembled powerful workflows: the codebase-archaeologist subagent, Plan Mode exploration loops, `/batch` for parallel analysis, hooks for quality enforcement. But these live in your head and in these digests. **Skills turn them into permanent, version-controlled slash commands** that anyone on your team can invoke with a single `/command`. For a consultant, this is the difference between personal productivity and a scalable capability.

**What Skills are:**
A Skill is a markdown file in `.claude/skills/` (project-level, shared via git) or `~/.claude/skills/` (personal). Each file becomes a `/slash-command` that loads specific instructions into Claude's context. Unlike CLAUDE.md (which is always loaded), Skills are loaded on demand — either when you type the command or when Claude auto-detects relevance.

**The anatomy of a skill:**

```markdown
# .claude/skills/analyze-architecture/SKILL.md
---
name: analyze-architecture
description: Reverse-engineers architecture documentation from a legacy codebase. Use when the user asks to analyze, document, or map a codebase's structure.
---

You are a senior software architect specializing in legacy system analysis.

## Workflow
1. Enter Plan Mode — explore the codebase structure without modifying anything
2. Identify the top 5-10 end-to-end business flows
3. For each flow, generate a Mermaid `flowchart LR` diagram with repos as swimlanes
4. Document all API endpoints with request/response schemas
5. Map data structures and their relationships
6. Flag architectural risks and technical debt

## Output Format
- Structured markdown with Mermaid diagrams
- Summary table of services, their responsibilities, and integration points
- Risk register with severity ratings
```

**Three key features to know:**

1. **Auto-triggering**: If the `description` contains trigger phrases Claude recognizes in your prompt, the skill loads automatically. Write descriptions that start with an action verb and include the keywords your requests naturally contain.

2. **`disable-model-invocation: true`**: Add this to skills with side effects (`/deploy`, `/send-report`). This ensures the skill only runs when YOU explicitly type the command — Claude won't auto-trigger it.

3. **Supporting files**: Skills can include a directory of supporting files (templates, schemas, example outputs) that Claude references when executing the skill. Put them alongside `SKILL.md` in the skill's directory.

**For the ISP presentation:** A `.claude/skills/` directory committed to a project repo is a **concrete artifact** demonstrating GenAI-in-SDLC integration. Show a slide with 3-4 skill files and explain: "These are reusable AI workflows that any developer on the team can invoke. They encode our engineering standards and produce consistent output."

### Key Takeaway
**Skills are how you turn 21 digests of learning into a team asset.** Create project-level skills in `.claude/skills/` for workflows your team shares (committed to git), and personal skills in `~/.claude/skills/` for your own consulting toolkit. The Quick Win below gives you your first one — the reverse engineering workflow from the past week, packaged as a single `/analyze-architecture` command.

## Resource 2: 1M Context Window — New Rules for the 100K-Line Project

**Source:** [Claude Code 1M Context Window: What It Means for Your Workflow — claudefa.st](https://claudefa.st/blog/guide/mechanics/1m-context-ga)
**Also:** [Claude Code 1M Context: What It Unlocks for Large Codebases — Verdent Guides](https://www.verdent.ai/guides/claude-code-1m-context-window)
**Also:** [Best Practices for Claude Code — Claude Code Docs](https://code.claude.com/docs/en/best-practices)
**Time:** ~5 min read
**Category:** Large Codebase Analysis & Performance

### Why This Is Relevant
Your Claude Code sessions now run on Opus 4.6 with a **1M token context window** — roughly 830K usable tokens after system prompt overhead. This fundamentally changes the approach to the 100K-line Java/Python reverse engineering project. Previous digests recommended aggressive subagent delegation to manage context limits. With 1M tokens, the calculus shifts: you can often **load entire service boundaries directly** instead of delegating everything to isolated subagents.

**The new decision framework — when to load directly vs. delegate:**

| Scenario | Approach | Why |
|---|---|---|
| Analyzing one service (< 200 files) | **Load directly** in main context | 1M tokens can hold it all; full visibility produces better analysis |
| Cross-referencing 2-3 services | **Load directly** with both services | See the integration points without summarization loss |
| Full codebase scan (100K+ lines) | **Subagents + /batch** | Still exceeds 1M; delegate to specialized agents per service boundary |
| Long debugging session | **Compact after resolution** | Failed attempts waste context; `/compact` after fixing preserves the solution |
| Multi-day project | **CLAUDE.md + fresh sessions** | Persist findings in CLAUDE.md; start each day with a clean 1M window |

**Three strategies that change with 1M:**

1. **Load the test alongside the implementation.** Previously, you'd read the source file and describe the test expectations. Now, load both — Claude sees the full picture and produces more accurate analysis. For the Java/Python project, load the service code AND its test suite simultaneously.

2. **Stop pre-filtering context.** With smaller windows, you had to carefully choose which files to show Claude. With 1M, err on the side of loading more. Include the config files, the build files, the READMEs. The marginal cost of extra context is low; the cost of Claude missing a crucial detail is high.

3. **Compact strategically, not aggressively.** The old rule was "compact early, compact often." With 1M, compact only at natural breakpoints — when you finish one subtask and move to the next. Use `/compact Focus on the architecture findings from the Java services` to preserve what matters.

**The subagent sweet spot with 1M:**
Subagents remain valuable not for context limits but for **context cleanliness**. Even with 1M tokens, a long debugging session pollutes context with failed attempts. Delegate exploratory or trial-and-error work to subagents so your main context stays focused on decisions and results.

### Key Takeaway
**For the 100K-line project: load service boundaries directly (they'll fit), use subagents for exploration and trial-and-error (to keep main context clean), and use `/batch` for codebase-wide operations (where the full codebase exceeds even 1M).** The 1M window doesn't eliminate subagents — it changes when you need them. They're now a cleanliness tool, not a survival tool.

It's also essential prep for the April 14 Office Hours: when Anthropic demos features, you'll understand the model capabilities behind them.

**The 4 improvements that matter most for your work:**

1. **Better planning for multi-step tasks**: Opus 4.6 "plans more carefully" before executing. This means Plan Mode (Shift+Tab) is even more effective — the model's internal planning capabilities align with the explicit planning workflow. For the reverse engineering project, Claude will produce more structured, comprehensive analysis plans before diving in.

2. **Sustained concentration on long agentic tasks**: The model maintains focus over longer sessions without degrading. This directly benefits `/batch` operations and Agent Teams — each worker agent stays on task for longer without drifting. If your April 10 `/batch` attempts felt more reliable than earlier sessions, this is why.

3. **Improved code review and self-correction**: Opus 4.6 catches its own mistakes more reliably. When combined with the `/simplify` command (which asks Claude to review its own changes), you get a stronger feedback loop. For client deliverables, this reduces the "second pair of eyes" burden.

4. **128K max output tokens**: Up from previous limits, Opus 4.6 can generate much larger outputs in a single response. For architecture documentation, this means Claude can produce a complete service-by-service analysis without truncation. Previously, you might need to ask "continue" multiple times for large documents — now it can deliver the full analysis in one pass.

### Key Takeaway
**Opus 4.6's improvements are force multipliers for features you're already using.** Better planning makes Plan Mode stronger. Sustained concentration makes `/batch` and Agent Teams more reliable. Self-correction makes `/simplify` more thorough. 128K output means complete architecture documents in one pass. You don't need to change your workflow — the model upgrades amplify what you're already doing. ## Quick Win: Create Your First Consulting Skill — `/reverse-engineer` (3 min)

Package the entire reverse engineering workflow from the past 4 digests into a single slash command your team can use:

**Step 1:** Create the skill directory and file:

```bash
mkdir -p .claude/skills/reverse-engineer
```

**Step 2:** Create `.claude/skills/reverse-engineer/SKILL.md`:

```markdown
---
name: reverse-engineer
description: Reverse-engineers architecture documentation from a legacy codebase. Use when the user asks to analyze, document, reverse-engineer, or map a codebase's architecture, API flows, or data structures.
---

You are a senior software architect specializing in legacy system analysis and documentation recovery.

## Phase 1: Explore (Plan Mode)
- Enter Plan Mode (do not modify any files)
- Map the top-level directory structure and identify service boundaries
- Identify languages, frameworks, and build systems in use
- List all entry points (main classes, API routers, CLI handlers)
- Propose the top 5-10 end-to-end business flows to document

## Phase 2: Analyze (Subagents)
For each service boundary, spawn a subagent to:
1. Document all API endpoints (REST, SOAP, message queues) with request/response schemas
2. Map data structures and their relationships
3. Trace business flows using Mermaid `flowchart LR` diagrams (one swimlane per service)
4. Identify external dependencies and integration points
5. Flag architectural risks, anti-patterns, and technical debt

## Phase 3: Synthesize
- Aggregate subagent findings into a single architecture document
- Create a system overview diagram showing all services and their relationships
- Produce a risk register sorted by severity
- Generate a summary suitable for stakeholder presentation

## Output Format
- Structured markdown with Mermaid diagrams
- Each service gets its own section with: purpose, API surface, data model, flows, risks
- Executive summary at the top (3-5 bullet points for non-technical stakeholders)
```

**Step 3:** Commit it to your project repo so the whole team gets it:

```bash
git add .claude/skills/
git commit -m "Add /reverse-engineer skill for architecture analysis"
```

Now anyone on the team types `/reverse-engineer` and gets the full methodology — Plan Mode exploration, subagent delegation, flow-first documentation, Mermaid diagrams — all encoded in one command.

**Why this compounds:** This skill is a **consulting deliverable template**. Once validated on the 100K-line Java/Python project, it becomes a reusable asset for any codebase takeover engagement. Modify the skill for each client's tech stack, commit it to their repo, and the team has an instant architecture analysis workflow. This is the "productize your approach" pattern that turns project work into capability building.
