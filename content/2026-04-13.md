## Resource 2: Prompt & Agent Hooks — Build Self-Verifying AI Workflows

**Source:** [Automate Workflows with Hooks — Claude Code Docs](https://code.claude.com/docs/en/hooks-guide)
**Also:** [Claude Code Hooks Reference: All Events — Pixelmojo](https://www.pixelmojo.io/blogs/claude-code-hooks-production-quality-ci-cd-patterns)
**Also:** [Claude Code Hooks: Complete Guide — aiorg.dev](https://aiorg.dev/blog/claude-code-hooks)
**Time:** ~5 min read (focus on prompt/agent hooks sections)
**Category:** Quality Automation & SDLC Integration

### Why This Is Relevant

Previous digests covered hooks basics — `PreToolUse` for blocking, `PostToolUse` for formatting, `Notification` for alerts. But Claude Code now supports three advanced hook types that turn hooks from simple gatekeepers into **intelligent quality verification systems**. This is the missing piece for your ISP GenAI-in-SDLC story: AI that verifies its own work before declaring it done.

**The three hook types beyond shell commands:**

1. **`"type": "prompt"`** — Sends hook context to a Claude model (Haiku by default) for a single-turn yes/no judgment. The model returns `{"ok": true}` to proceed or `{"ok": false, "reason": "..."}` to block. Perfect for semantic checks that regex can't express: *"Does this edit touch authentication logic?"* or *"Are all requested tasks complete?"*

2. **`"type": "agent"`** — Spawns a full subagent with tool access (file reading, code search, command execution) to verify conditions. Up to 50 tool-use turns with a 60-second default timeout. Use when verification requires inspecting actual codebase state, not just the hook input.

3. **`"type": "http"`** — POSTs event data to a URL endpoint. Useful for centralized audit logging, external quality gates (SonarQube, Snyk), or team-wide policy enforcement via a shared service.

**The killer pattern — self-verifying Stop hook:**

This is the most powerful combination. A `Stop` hook fires whenever Claude finishes responding. Pair it with a prompt or agent hook, and Claude **verifies its own work before stopping**:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Check if all requested tasks are complete and tests pass. If not, respond with {\"ok\": false, \"reason\": \"what remains to be done\"}."
          }
        ]
      }
    ]
  }
}
```

If the verification model returns `"ok": false`, Claude **keeps working** using the `reason` as its next instruction. This creates a feedback loop where the AI agent cannot declare success until an independent check confirms it.

**Important safety note:** Add a `stop_hook_active` guard to prevent infinite loops:

```bash
#!/bin/bash
INPUT=$(cat)
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0  # Allow Claude to stop on the verification pass
fi
# ... verification logic
```

**For the ISP presentation:**

This is your strongest SDLC integration story. Frame it as: *"The AI doesn't just generate code — it verifies its own output against quality gates before a human ever sees it. The Stop hook ensures tests pass, style checks clear, and all requested changes are complete. This is deterministic quality enforcement on AI-generated code."* Enterprise CTOs care about governance and quality control — this pattern directly addresses both.

### Key Takeaway

**Prompt and agent hooks at the `Stop` event create self-verifying AI workflows.** This is the pattern that makes Claude Code enterprise-ready: the AI checks its own work using an independent model evaluation before stopping. For the ISP presentation, this is your "governance and quality" slide. For your own workflow, add a Stop prompt hook to your reverse engineering skill to ensure all API endpoints are documented before Claude declares the analysis complete.

## Quick Win: Run `/team-onboarding` and Add a Stop Verification Hook (3 min)

**Part 1 — Run `/team-onboarding` (30 seconds):**

Open Claude Code and type `/team-onboarding`. Review the generated guide — it shows your actual usage patterns packaged as a teammate onboarding doc. Save or screenshot the output for tomorrow's Office Hours and the ISP presentation.

**Part 2 — Add a self-verifying Stop hook (2 min):**

Add this to your `~/.claude/settings.json` to make Claude verify task completion before stopping:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Review whether Claude completed all tasks the user requested. Check: (1) all requested changes were made, (2) no obvious errors in the output, (3) if tests were mentioned, they were run. If anything is incomplete, respond with {\"ok\": false, \"reason\": \"description of what remains\"}. If everything looks complete, respond with {\"ok\": true}."
          }
        ]
      }
    ]
  }
}
```

This takes 2 minutes to set up and immediately improves every future Claude Code session. Claude will self-check before stopping, catching incomplete work that you'd otherwise have to notice yourself. Combined with your `/reverse-engineer` skill, this means the architecture analysis will verify all endpoints are documented before declaring done.

**Why both of these compound:** `/team-onboarding` shows how you use Claude Code. The Stop hook shows how Claude Code governs itself. Together, they tell the enterprise story: *"AI that onboards your team AND verifies its own quality."* That's tomorrow's Office Hours pitch in one sentence.
