## Resource 1: Plan Mode — The Explore-Plan-Execute Loop

**Source:** [Claude Code Plan Mode: Design Review-First Refactoring Loops — DataCamp](https://www.datacamp.com/tutorial/claude-code-plan-mode)
**Time:** ~5 min read
**Category:** Structured Problem-Solving

### Why This Is Relevant
As a consultant, you don't jump straight to recommendations — you define the problem, explore the data, form a hypothesis, and then act. Plan Mode enforces exactly this discipline on Claude Code. It locks Claude into **read-only mode** (Shift+Tab): Claude can read files, search the codebase, and ask you questions, but it *cannot* write, edit, or execute anything. Only after you review and approve the plan do you switch back to normal mode for execution.

**When to use Plan Mode (the 3-file rule):**
- Any change that touches **3 or more files**
- Refactors that move code between modules
- Analysis tasks where you need Claude to understand the full picture before acting
- Any work on a client project where mistakes are costly

**The Explore-Plan-Execute workflow:**
1. **Explore** (Shift+Tab to enter Plan Mode): Ask Claude to read the relevant files, understand the structure, and surface questions. Claude maps the territory without touching anything.
2. **Plan** (Ctrl+G to edit): Claude proposes a step-by-step plan. You review, adjust, add constraints. This is your "hypothesis alignment" step.
3. **Execute** (Shift+Tab to exit Plan Mode): Claude follows the approved plan. Because it already understands the codebase, execution is faster and more accurate.

**For large refactors:** Break the work into sequential plans rather than one giant one. Finish one plan, commit, then start a fresh session for the next phase. Use `/rename` to label each session (e.g., "client-x-data-pipeline-refactor-phase-1") so you can resume with `/resume`.

### Key Takeaway
**Next time you start a complex task, hit Shift+Tab first.** Spend 2 minutes in Plan Mode letting Claude explore and propose an approach before it writes a single line. This mirrors the consulting "structure before solution" discipline and consistently produces better results — fewer bugs, fewer rework loops, and a plan you can explain to stakeholders.

## Resource 2: How Claude Code Builds Its System Prompt — Understanding the Machine

**Source:** [How Claude Code Builds a System Prompt — Drew Breunig](https://www.dbreunig.com/2026/04/04/how-claude-code-builds-a-system-prompt.html)
**Time:** ~5 min read
**Category:** Meta-Learning & Prompt Mastery

### Why This Is Relevant
You've been writing CLAUDE.md files, configuring hooks, and crafting prompts for weeks. But do you know *how* Claude Code actually assembles all of this into the prompt that drives each session? Understanding the internals turns you from a "recipe follower" into someone who can debug and optimize your setup.

**The key insights:**

1. **It's not one prompt — it's ~80 modular pieces** assembled into a unique prompt per session through a priority layer system. Your CLAUDE.md, MCP tool descriptions, language settings, and environment info are all injected at different layers.

2. **The 6 priority layers** (highest to lowest): Override → Coordinator → Agent → Custom → Default → Append. Your CLAUDE.md instructions live in the Append layer (lowest priority), which means they can be overridden by higher layers. This is why sometimes Claude "ignores" your CLAUDE.md — it's not ignoring it, a higher-priority instruction is winning.

3. **The cache boundary trick**: The system prompt is split by a marker (`__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__`) into static (shared across all users) and dynamic (per-session) halves. The static half is cached across all Claude Code users on the same version — this is why Claude Code feels fast. Your per-user instructions are injected via `<system-reminder>` tags in messages, not the system prompt itself.

4. **Why this matters for your CLAUDE.md**: Keep your CLAUDE.md concise and factual (under 200 lines). Don't try to override Claude's core behavior — the higher priority layers will win. Instead, focus on project-specific context: directory structure, coding conventions, build commands, and links to docs.

### Key Takeaway
**Your CLAUDE.md is not fighting against the system prompt — it's layered on top of it.** Write it like a project onboarding doc for a smart colleague, not like a set of override commands. Focus on *what's unique about this project* rather than trying to change Claude's general behavior. Understanding this architecture explains why short, factual CLAUDE.md files outperform long, directive ones.

## Resource 3: Automated PR Reviews with Claude Code GitHub Actions

**Source:** [Claude Code × GitHub Actions: Automate PR Reviews, Issue Handling, and Test Generation — Claude Lab](https://claudelab.net/en/articles/claude-code/claude-code-github-actions-automated-workflow)
**Time:** ~4 min read
**Category:** Quality Automation & CI/CD

### Why This Is Relevant
Previous digests covered hooks for local enforcement and sub-agents for parallel work. But what about quality control on code that's already been pushed? The official `anthropics/claude-code-action` GitHub Action runs Claude Code inside your CI/CD pipeline — it reads every PR diff, understands the full repo context, and posts specific, actionable review comments before any human reviewer sees the code.

For a consultant who ships code to clients or maintains analysis pipelines, this is an automated senior reviewer that catches bugs, security issues, and style violations on every push.

**Setup in 2 minutes:**
1. Run `/install-github-app` inside Claude Code — it walks you through the GitHub App setup and required secrets
2. Or manually add the workflow file:

```yaml
name: Claude PR Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
```

**What it catches:**
- Logic bugs and edge cases in the diff
- Security vulnerabilities (injection, auth issues)
- Code quality and style violations
- Missing tests or documentation
- You can interact with Claude directly in the PR thread — ask follow-up questions, request fixes, or have it generate a fix PR

**Cost:** A typical review costs a few cents using Sonnet. For a consulting team reviewing 5-10 PRs per day, that's under $1/day for automated senior-level review.

### Key Takeaway
**Run `/install-github-app` on your most active repo today.** Within minutes, every PR gets an automated review that catches issues before human reviewers spend time on them. This is especially valuable for consulting engagements where you're shipping code to clients — an extra layer of quality assurance that runs 24/7 and costs almost nothing.

## Quick Win: Try Plan Mode on Your Next Task (30 seconds)

The next time you open Claude Code to work on something non-trivial, start with this:

1. **Press `Shift+Tab`** — you'll see the mode indicator change to "Plan"
2. **Describe what you want to accomplish** — e.g., "I need to refactor the data processing pipeline to handle the new client CSV format"
3. **Let Claude explore** — it will read files, ask clarifying questions, and propose a plan
4. **Review the plan** — adjust with `Ctrl+G` if needed
5. **Press `Shift+Tab` again** to switch to execution mode — Claude follows the plan

That's it. You'll notice Claude's execution is faster and more accurate because it understood the full context before writing any code. This is the single highest-impact workflow change from this entire digest series — it works for code refactors, data analyses, report generation, and any multi-step task.

**Why this compounds:** Plan Mode + CLAUDE.md (from April 6) + hooks (from April 7) = a workflow where Claude understands your project (CLAUDE.md), thinks before acting (Plan Mode), and enforces quality automatically (hooks). These three features together represent the "consulting-grade" Claude Code setup.
